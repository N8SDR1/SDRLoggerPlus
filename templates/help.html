SDRLogger+ Quick Start Guide

v{{ version }}  ·  Ham Radio Logging · DX Cluster · TCI / Thetis SDR Integration · Solar Conditions · Rotator Control · CW Keyer / Decoder

Overview

SDRLogger+ is a browser-based ham radio contact logger designed for use with Expert Electronics TCI/SDR compatible rigs (EESDR, Anan, Hermes-Lite transceivers) via the TCI (Transceiver Control Interface) protocol, with full waterfall integration. It provides real-time frequency, mode and callsign synchronisation between your radio and the logging form, a live DX cluster display, distance mapping, ADIF import/export, and solar propagation data — all in a single dark-themed interface. SDRLogger+ also supports any HamLib-compatible rig for frequency/mode sync and DX spot tuning.

SDRLogger+ also features automatic digital mode QSO logging — QSOs completed in WSJT-X, JTDX, MSHV, VarAC, and other digital applications are captured automatically over UDP or TCP and saved to your log without any manual entry. An integrated Rotator Control panel can command your antenna rotator via PstRotator, GS-232, or EasyComm II whenever a DX station is looked up, automatically pointing your beam at the short-path or long-path bearing. The built-in CW Keyer / Decoder popup lets you send CW text via TCI or HamLib, decode incoming CW from TCI audio or any sound device, and fire programmable macro buttons with live {CALL}, {NAME}, and {RST} substitution from the QSO entry form.

First Run Setup

1 — Configure Your Station

Click the ⚙ Settings button in the top-right corner and fill in:

  • My Callsign — your callsign (e.g. N8SDR). This is sent to the DX cluster as your login and displayed in the header.
  • Operator Name — your first name, used for cluster login prompts.
  • Grid Square — your Maidenhead locator (e.g. EN82). Required for the distance map and beam headings.
Your callsign must be set before connecting to any DX cluster.

2 — TCI / Thetis Connection

Enter your Thetis TCI host (default 127.0.0.1) and port (default 50001). When connected, the TCI LIVE badge lights up in the header and the logging form auto-fills frequency, mode, and spotted callsigns.

If you change the TCI host or port in Settings and click Save Settings, SDRLogger+ automatically drops and re-establishes the TCI connection — no restart required.

3 — Callsign Lookup (QRZ or HamQTH)

SDRLogger+ supports two callsign lookup services. Configure one or both — QRZ is tried first, with HamQTH used automatically as a fallback if QRZ is not configured or the callsign is not found.

QRZ.com

Enter your QRZ.com username and password. An active QRZ XML Data subscription ($35.95/yr) is required for full lookups. When a callsign is entered, SDRLogger+ automatically fills the Name and QTH fields and updates the distance map.

To upload each saved QSO automatically to your QRZ online logbook, enter your QRZ Logbook API Key (found under qrz.com → My Logbook → Settings → API Key). The Auto-upload QSOs to QRZ Logbook checkbox controls whether uploads fire — it is ON by default. Uncheck it to disable uploads while keeping your key saved.

HamQTH.com

HamQTH.com offers a free callsign lookup service — no subscription required. Register for a free account at hamqth.com, then enter your username and password in Settings → HamQTH Lookup. Click Test Login to verify your credentials.

HamQTH returns name, QTH, grid square, country, CQ zone, and ITU zone — the same fields used to fill the QSO entry form and update the distance map.

If you have neither QRZ nor HamQTH configured, the callsign lookup button and auto-fill will not activate. HamQTH is recommended for operators who do not have an active QRZ XML subscription.

ARRL LoTW (Logbook of the World)

SDRLogger+ can automatically upload each saved QSO to ARRL Logbook of the World — the ARRL's QSO confirmation and award credit system used by operators worldwide.

LoTW uses digital certificate signing rather than an API key. Uploads are handled by the TQSL (TrustedQSL) application which must be installed separately and configured with your LoTW callsign certificate before SDRLogger+ can upload on your behalf. To set up:

  1. Install TQSL from lotw.arrl.org and complete the LoTW account and certificate request process
  2. In TQSL, create a Station Location (File → Station Locations) — give it a memorable name like Home-QTH
  3. In SDRLogger+ ⚙ Settings → LoTW, enter the full path to tqsl.exe (typically C:\Program Files (x86)\TrustedQSL\bin\tqsl.exe), or just tqsl if TQSL is on your Windows system PATH
  4. Enter your Station Location name exactly as it appears in TQSL. If left blank, SDRLogger+ falls back to your callsign certificate
  5. Click 🔍 Verify TQSL Path to confirm SDRLogger+ can find the executable
  6. Check Auto-upload QSOs to LoTW and click Save Settings

Once enabled, each QSO is written to a temporary ADIF file and passed to TQSL with the flags -d -u -x -l <Station Location>, which signs the record with your certificate and uploads it to LoTW in the background. The QSO save confirmation message shows LoTW ✓ on success or a brief error if the upload fails. For full technical details on the submission protocol see the LoTW developer documentation ↗.

Club Log

SDRLogger+ can upload each QSO in real-time to Club Log — the popular online tool for DXCC tracking, contest logging, and log verification used by operators worldwide.

Club Log uses its own credential system — you will need three things from your Club Log account:

  • Email — the email address registered with Club Log
  • Application Password — a separate password for API access, set in your Club Log account settings under Application Password. This is not your regular login password
  • Developer API Key — a per-application key that identifies SDRLogger+ to the Club Log API. Request yours at clublog.org/requestapikey.php

SDRLogger+ uses the Club Log real-time API (realtime.php), which is the correct endpoint for single-QSO uploads as you log. The batch endpoint (putlogs.php) is for bulk ADIF imports only — using it for repeated single-QSO uploads triggers automatic IP blocking and must not be used here.

⚠ Important — IP blocking: Club Log automatically firewalls IP addresses that send repeated failed authentication requests. Always click 🔑 Test Credentials and confirm success before enabling auto-upload. If you change your Application Password in Club Log, update it in SDRLogger+ Settings immediately — uploads are automatically halted after a single 403 error to protect your IP.
LoTW upload is off by default. It only activates once you have checked Auto-upload QSOs to LoTW in Settings and saved. QRZ Logbook and LoTW uploads are independent — you can enable both, either, or neither.

PATH note: If you enter just tqsl as the path (without a full directory), TQSL must be added to your Windows Environment Variables → Path (System Properties → Advanced → Environment Variables). If you use the full path to tqsl.exe this is not required.

HamLib Rig Control

SDRLogger+ can connect to any HamLib-compatible transceiver via the rigctld network daemon — giving non-TCI rig owners full frequency/mode following and DX spot tuning. Over 300 radios are supported including Yaesu FT-991A, FT-DX10, Icom IC-7300, IC-7610, IC-705, Kenwood TS-590, TS-890, and many more.

How It Works

HamLib's rigctld program runs on your PC and acts as a small TCP server on port 4532. SDRLogger+ connects to it as a client, polling for frequency and mode updates every 1.5 seconds and sending tune commands when you click a DX spot.

TCI and HamLib can both be enabled at the same time. TCI always takes priority — HamLib only updates the form and tunes the rig when TCI is not connected. This lets TCI SDR users enable HamLib as a fallback without any conflict.

Step 1 — Install HamLib

Download the Windows binary package from the official HamLib download page: github.com/Hamlib/Hamlib/wiki/Download ↗ (direct Windows builds also available at SourceForge ↗). Extract the zip anywhere — no installer needed. The key executable is rigctld.exe.

Step 2 — Find Your Rig Model Number

Run the following in a command prompt to list all supported rigs and find your model number:

rigctl.exe -l

Common model numbers:

Step 3 — Start rigctld

Open a command prompt and run your rigctld command, for example:

rigctld.exe -m 3073 -r COM3 -s 19200 -T 127.0.0.1 -t 4532

Leave this window open — rigctld must keep running while SDRLogger+ is in use. You can create a .bat file to launch it automatically alongside SDRLogger+.

Step 4 — Enable in SDRLogger+

  1. Go to ⚙ Settings → HamLib / rigctld
  2. Check Enable HamLib rig control
  3. Set Host to 127.0.0.1 and Port to 4532 (defaults are correct if rigctld is on the same PC)
  4. Click Test Connection to verify — a green confirmation message confirms rigctld is reachable
  5. Click Save Settings

Once connected, the RIG LIVE badge appears in the header. Frequency and mode will begin following your rig, and clicking any DX cluster spot will tune your radio automatically.

DX Cluster

TCILog supports up to 3 simultaneous Telnet DX cluster connections plus the Spothole.app REST API aggregator (POTA, SOTA, RBN, clusters — no API key required). All sources are merged into a single spot list.

Connecting

Configure cluster servers in Settings (DX Clusters section), then click CONNECT in the DX Cluster panel. The status dot turns green when connected.

Spot List Controls

  • Band filter — automatically tracks your TCI VFO frequency and filters to the active band.
  • Age filter — dropdown (5/10/15/30 min) prunes old spots.
  • Max spots — number input (1–100) limits how many spots are displayed at once (default 25).
  • Sort — click column headers (DX Call, Mode, Freq kHz) to sort ascending/descending.

Advanced Spot Filters (Settings → DX Clusters)

  • Mode Filter — comma-separated modes to show (e.g. FT8,CW,SSB). Leave blank for all modes.
  • DXCC Entity Filter — comma-separated callsign prefixes (e.g. W,VE,DL,JA). Only spots whose DX callsign starts with one of these prefixes are shown. Leave blank for all entities.
  • CQ Zone Filter — comma-separated zone numbers (e.g. 4,5,14). Leave blank for all zones.
  • Continent Filter — comma-separated continent codes (e.g. NA,EU,AS). Uses prefix-based continent lookup. Supported codes: NA North America, SA South America, EU Europe, AF Africa, AS Asia, OC Oceania. Leave blank for all.

Hot Stations (Watchlist Highlight)

Enable Highlight Hot Stations in Settings → DX Clusters and enter a comma-separated watchlist of callsigns (e.g. DX0DX,VK9XX,ZL9HR). When a watched callsign appears in the spot list, it is highlighted with a red callsign and a red left border — making it easy to spot rare DX at a glance. Toggle the checkbox to turn highlighting on or off without clearing the watchlist.

Clicking a Spot

Clicking any spot row fills the callsign and frequency in the logging form and sends a TCI tune command to retune your Thetis radio to that frequency and mode automatically.

Panadapter Integration

All received spots are mirrored to the Thetis or EESDR panadapter as colored markers. The marker color is configurable in Settings → Display → Panadapter Spot Color.

The Rotator Panel

The 🧭 Rotator Control panel appears in the right column and shows:

PstRotator Setup Notes

SDRLogger+ uses PstRotator's documented UDP external control interface on port 12000:

  • Azimuth commands (<PST><AZIMUTH>270</AZIMUTH></PST>) are sent via UDP to port 12000
  • Live AZ is polled every 500 ms — SDRLogger+ sends <PST>AZ?</PST> to port 12000 and PstRotator replies on port 12001 (port + 1) with AZ:xxx.x
  • The Stop command (<PST><STOP>1</STOP></PST>) is also sent via UDP — the same method used by N1MM Logger+, Swisslog, and Logger32
In PstRotator, go to Communication → UDP Control Port and ensure the port is set to 12000 (matching SDRLogger+ Settings). No TCP server setup is needed.

  • Enable/disable the rotator
  • Select protocol (PstRotator, GS-232, or EasyComm II)
  • Set host (default 127.0.0.1) and port (default 12000)
Use the Test button in the Rotator panel to verify the connection before operating. The status dot turns green on a successful command.

CW Keyer / Decoder

The CW Keyer / Decoder opens as a separate popup window — click the CW button in the SDRLogger+ header bar. It can be moved to any monitor. All settings are saved automatically and restored the next time you open it.

Opening the Window

  • Click CW in the top-right header of SDRLogger+
  • The popup is independent — SDRLogger+ continues running normally while the CW window is open
  • Connection status (TCI / HamLib / Disconnected) is shown in the top-right corner of the CW window

Sending CW

  • Type your message in the Send text field and press Enter or click SEND
  • Click STOP at any time to abort transmission mid-message
  • Set transmit speed with the TX WPM slider (5–60 WPM)
  • Sent text appears in the TX Sent monitor window for confirmation
  • Click the red CLR button below the TX Sent window to clear it
  • Auto-Log 73 toggle — when enabled, sending any message containing 73 automatically fires ▶ Save QSO in SDRLogger+, logging the contact without needing to click it manually. The status bar confirms with ✓ QSO auto-logged. Keep this OFF during casual monitoring or when testing macros.
Thetis must be in CW mode (CWL or CWU) for the send command to work. If the radio is in SSB or any other mode, Thetis will silently ignore the CW text. Switch your VFO to a CW mode first.

How CW is Keyed

SDRLogger+ routes CW through the best available path automatically:

  • TCI connected — sends cw_macros:0,text directly to Thetis via the TCI WebSocket. Thetis handles all timing and keys the radio hardware (HL2, ANAN) via its internal CWX engine. No COM port or PTT interface needed.
  • HamLib connected (no TCI) — sends \send_morse text to rigctld, which keys the rig via CAT/serial
Break-in (QSK) is not a TCI command — it is configured directly in Thetis Setup → CW and applies to all CW transmitted through Thetis regardless of source.

CW Decoder

The decoder uses IQ downconversion — sample-by-sample quadrature mixing — to track the carrier envelope at up to ~1400 updates per second. A running matched filter (~0.5 dit long) smooths sub-dit noise spikes without blurring element edges. Detection uses two gates in series: Squelch gates the decoder off entirely until the signal-to-noise ratio is sufficient; within that gate Threshold sets the exact mark/space boundary with asymmetric hysteresis (+6% to start a mark, −12% to hold it) that prevents chatter on marginal signals. Each element is classified by a Bayesian Gaussian timing model that learns the dit and dah distributions for the specific operator being copied — no fixed ratio is assumed. The model bootstraps from the first few elements (using your TX WPM as the starting estimate) then switches to full Bayesian classification. Speed tracking is asymmetric: the model follows speed-ups quickly and slowdowns more cautiously, matching the way real operators accelerate. Inter-element spacing is tracked independently because operators consistently send gaps slightly shorter than a full dit. The noise floor updates asymmetrically — freely during silence, fall-only during marks — so strong signal energy never contaminates the noise reference. Pressing CLR clears decoded text but intentionally preserves the learned timing so the decoder does not have to re-adapt to the same station. Decoded text appears in the RX Decode window. Both windows are resizable — drag the bottom edge.

Quick Start — Getting a Decode

  1. Select your audio source from Audio In and click Start Listening
  2. Watch the waterfall — a CW signal appears as a bright vertical streak. If you see one, click it to tune the decoder to that frequency (⌖ Snap mode will lock to the peak automatically)
  3. Confirm the Sig bar pulses on each dit and dah. If it does not, the Tone Hz or AFC is not centred on the signal — click the streak in the waterfall again
  4. The first few characters use a timing bootstrap; full Bayesian decoding kicks in after ~8 elements and accuracy improves from there
  5. If copy looks good but speed is wrong, enable Auto-match RX WPM and set reasonable min/max clamps (e.g. 10–40 WPM)

Troubleshooting Decodes

  • Sig bar flat / no output — Squelch may be too high (lower it toward ×1.1), or the tone frequency is off. Look at the waterfall: the signal streak should be under the yellow cursor line. Click the streak to re-tune.
  • Garbage characters / noise triggers — Squelch too low or Threshold too low. Raise Squelch first; if the band is noisy, try ×2.0–×2.5. Then raise Threshold slightly (30–35%) if false marks persist.
  • Decoding stops after running for a while — click the signal in the waterfall to force a full reset and immediate re-calibration. This is equivalent to a Stop/Start but without losing your audio device selection.
  • Changed frequency and decoding died — move the Tone slider to the new sidetone pitch (or click the new signal in the waterfall). The full signal chain resets automatically on any tone change.
  • Dits read as dahs or vice versa — the model is still bootstrapping. Give it a full callsign exchange (~10–15 elements) to adapt. Ensure TX WPM is set close to the station's actual speed before you start listening.
  • AFC jumps to wrong signal — disable AFC, click the correct signal in the waterfall using ⌖ Snap mode, then re-enable AFC. Or switch to ⌖ Exact click mode and click precisely on the desired carrier.

Controls Reference

  • Audio In — select your audio source from the dropdown:
    • TCI Audio — pulls RX audio directly from Thetis over the TCI WebSocket. No VAC or sound card routing needed. Thetis must be connected.
    • Windows audio inputs — any device Windows sees: VAC virtual cables, USB codecs, physical soundcards, loopback devices
  • Click Start Listening (next to the ↻ refresh button) to begin decoding; click again to stop
  • If devices appear as Audio Input () with no name, Windows is blocking the browser from reading device labels — see the tip below. Click the ↻ refresh button after fixing the permission to repopulate the list with proper names.
    • Why microphone access is required: The browser requires microphone permission to display the names of your audio input devices — even if you intend to use TCI audio or a VAC virtual cable rather than a real microphone. Granting access does not turn on your microphone and does not record or transmit anything; it simply allows the browser to list available audio devices so you can select the correct one. Your actual microphone will only be used if you explicitly select it in the Audio In dropdown. To grant access, go to Windows Settings → Privacy & Security → Microphone, scroll down, and make sure your browser (Chrome, Edge, Firefox) is toggled ON. Then click the ↻ refresh button in the CW window to repopulate the list. See also: Microsoft: Windows camera, microphone and privacy.
  • Tone Hz — set to match the CW sidetone pitch in Thetis (default 700 Hz). The AFC toggle beside the Tone slider switches Automatic Frequency Control on or off (default ON). When ON, AFC automatically tracks the carrier within the selected AFC range (±50 / ±100 / ±150 / ±200 Hz — use the adjacent dropdown to choose; default ±100 Hz). The indicator shows the locked frequency (e.g. AFC 703 Hz) so you can see it working. When OFF, the indicator shows AFC OFF and the decoder locks exactly to the Tone Hz you have set. Moving the Tone slider performs a full signal-chain reset — envelope state, matched filter, IQ oscillator, and decoder state are all cleared so the new frequency calibrates cleanly from scratch without needing a restart. AFC only locks when a genuine signal peak is found — it will not latch onto noise.
  • Waterfall — a scrolling spectrum display sits to the right of the two control columns (Tone/AFC on the left, Squelch/Threshold on the right), showing ±400 Hz around the set tone. New rows appear at the top; colour runs from black (no signal) through dark blue to white (strong signal). A red cursor line marks the live AFC-locked (or manually set) frequency with a Hz label. Frequency scale ticks appear along the bottom of the strip. The ⌖ Snap / ⌖ Exact toggle sits directly to the right of the canvas. Works with both TCI and microphone audio — no separate data source needed.
    • Click to tune — click anywhere on the waterfall while listening to re-tune the decoder to that frequency. A full signal-chain reset happens automatically so the new frequency starts calibrating immediately.
    • ⌖ Snap (default) — click mode that finds the strongest signal peak within ±60 Hz of your click and locks onto it. Useful when a carrier is visible in the waterfall but not exactly under your cursor.
    • ⌖ Exact — click mode that tunes to precisely where you click. Toggle between Snap and Exact using the ⌖ Snap / ⌖ Exact button; your preference is saved.
    • The waterfall is purely a display aid — it has no effect on decode quality and can be ignored entirely if not needed.
  • Squelch — sets the minimum signal-to-noise ratio required before the decoder runs at all. The decoder measures how much stronger the detected tone is relative to the background noise floor and only activates when that ratio meets this threshold. Range is ×1.1 to ×3.0, default ×1.5. Lower values (×1.1–×1.3) allow the decoder to work on weak or noisy signals but increase the chance of false triggers from band noise. Higher values (×2.0–×3.0) require a strong clean signal before decoding starts, which reduces garbage output on a busy or noisy band. Start at ×1.5 and adjust: lower if weak signals are not triggering the decoder at all; raise if noise is producing random characters when no signal is present.
  • Threshold — sets how far above the noise floor the signal must be to register as a mark. Slide left to lower the threshold (more sensitive — decodes weaker signals, but more noise risk). Slide right to raise it (more selective — strong signals only, cleaner output). Default 27%; minimum 15%. An asymmetric hysteresis band (+6% to start a mark, −12% to hold it) prevents chattering on signals hovering near the threshold.
  • Sig bar — shows real-time signal strength at the detected tone frequency. Should pulse clearly on each dit and dah. If the bar is completely flat while a signal is audible, the tone frequency is off — use the waterfall to click the correct signal.
  • RX WPM display — shows the decoder's current speed estimate (e.g. 22 WPM), updated after every classified element. Useful for confirming the model has locked onto the correct timing. Shows — WPM before any elements have been classified.
Recommended starting settings for most signals: Squelch ×1.5 · Threshold 27% · AFC ON · Snap mode ON. Lower Squelch toward ×1.1 for weak signals; raise toward ×2.5 for a noisy band. Threshold rarely needs to move off 27% — if you do adjust it, move in small steps (2–3% at a time).

Auto-Match RX WPM

When enabled, the TX WPM slider automatically tracks the speed of the station you are copying:

  • Enable the Auto-match RX WPM toggle in the Send panel
  • Set min and max clamp values to prevent wild swings (e.g. min 10, max 35 WPM)
  • Speed is derived from the Bayesian dit model — the same adaptive estimate used for element classification. Tracking is asymmetric: the model follows an operator speeding up quickly, but reacts more slowly to slowdowns, so a single hesitation does not throw off the estimate
  • A 2 WPM hysteresis prevents constant small TX speed adjustments between characters
  • The current measured receive speed is shown in the — WPM display next to the Sig bar and updates after every classified element

Macros

Sixteen programmable macro buttons are arranged in a 4×4 grid. Each has an editable label and message text.

  • Click a macro button to send it immediately
  • Right-click a macro button to open the macro editor — a modal dialog where you can type or paste the message text without triggering a send
  • Inside the editor, the token buttons at the top ({MYCALL}, {MYNAME}, {CALL}, {NAME}, {RST}, {NR}) insert the token at the cursor position — click any token button to insert it wherever your cursor is in the text field
  • Press Enter or click OK to save; press Esc or click outside the modal to cancel
  • Macro text and labels are saved automatically in your browser
  • Click RESET in the Macros panel header to restore all 16 buttons to factory defaults (confirmation required)

Supported substitution tokens — values are pulled live from settings and the QSO entry form:

TokenReplaced with
{MYCALL}Your own callsign (SDRLogger+ Settings → My Callsign)
{MYNAME}Your own operator name (SDRLogger+ Settings → Operator Name)
{CALL}Callsign of the station you are working (QSO entry Callsign field)
{NAME}Their name (QSO entry Name field — the station you are working)
{RST}My RST Sent — the signal report you are giving them (QSO entry My RST Sent field)
{NR}Current contest serial number (auto-increments after each send, persists across restarts)
Use {MYCALL} and {MYNAME} when identifying yourself (MY NAME IS {MYNAME} DE {MYCALL} K). Use {CALL} and {NAME} when addressing the other station. Use {RST} for the report you are sending them (UR RST {RST} {RST}).

Contest Serial Number ({NR})

The {NR} token inserts the current serial number, zero-padded to three digits (001, 002 … 999, 1000 …). It is designed for simple CW contests where an incrementing exchange number is required.

  • Enable the NR {NR} toggle in the Send panel to activate contest mode. When OFF, the token still expands but nothing increments and nothing is logged.
  • The current serial is displayed in the NR {NR} row; use + / − to manually adjust; CLR resets to 1 (with confirmation)
  • When the toggle is ON and any macro or typed message containing {NR} is sent successfully, the serial automatically increments to the next number
  • The sent serial number is appended to the Remarks field in the QSO entry panel (e.g. NR: 007) so it is logged with the contact
  • The serial number is saved to disk and survives restarts — you can close SDRLogger+ and resume a contest later without losing your count
  • A typical contest NR macro: NR {NR} {NR} TU DE {MYCALL} K

Display Options

  • Font size — Small / Medium / Large / X-Large dropdown — enlarges both the RX and TX windows
  • Text color — single color applies to both RX and TX windows: Green (default) / Amber / White / Cyan / Yellow-Green
  • All display preferences are saved and restored automatically

DX Spotting

SDRLogger+ can send DX spots directly from the logging form or from a recent log entry. Configure spotting under ⚙ Settings → DX Clusters → DX Spot Sending.

Spot Destination

A three-way segmented toggle controls where spots are sent. Only one destination is active at a time:

SettingWhat it does
C1Sends the standard DX FREQ CALLSIGN COMMENT command through your active Cluster 1 telnet connection. The cluster already knows your callsign from login — the spot propagates to the DX network normally.
OFFSpotting disabled. The 📡 Spot button is greyed out. Default setting.
SHSpothole is a read-only aggregator — it pulls spots from clusters, POTA, RBN, and SOTA, but has no public API for receiving externally submitted spots. Selecting SH will show a clear error. Use C1 to send spots.
If Cluster 1 is not connected when you click Spot, SDRLogger+ will automatically initiate the connection — click Spot again once the cluster connects.
Important — correct spot command format: SDRLogger+ sends DX FREQ DXCALL COMMENT (e.g. DX 14074.0 K1ABC FT8). Do not prefix with DX de YOURCALL — that is the broadcast format clusters use when sending spots out to users. Submitting a spot with that prefix causes clusters to silently ignore the command.

Confirm Before Spot

When Confirm before spot is checked (default ON), clicking the Spot button shows a one-line confirmation strip below the button row displaying the callsign, frequency, mode, and destination. Click SEND to transmit or Cancel to abort. Uncheck this option for single-click spotting during fast pileup or contest operating.

Spotting from the Entry Form

While a callsign is entered in the logging form, the 📡 Spot button uses the current form data — callsign, frequency, and mode. This is the primary path for spotting a station you are working or hearing.

Spotting the Last Logged QSO

After clicking ▶ Save QSO the form clears, but SDRLogger+ retains the just-logged callsign, frequency, and mode in a short-term buffer. The Spot button changes to 📡 Spot (W1AW) indicating it will spot the last logged station. This window is open for 10 minutes after logging — after that the buffer expires and the button returns to its idle state.

This means you can log a QSO first and spot it immediately after without re-entering any data.

Spotting from the Log Table

Right-click any row in the QSO log to access the context menu. The 📡 Spot this QSO option is available for QSOs logged within the last 10 minutes.

  • Within 10 minutes — option is active and clickable, respects Confirm setting
  • Older than 10 minutes — option is shown greyed out with the age displayed (e.g. "Spot — window expired (23 min ago)")
  • Destination set to OFF — option shows "enable destination in Settings"
The 10-minute limit on log-row spotting is fixed and cannot be changed. A DX spot is a real-time "this station is on the air now" signal — spotting old QSO data would send inaccurate information to the cluster network.

Spotter Region Filter

In addition to filtering by DX station continent, you can now filter the incoming spot list by where the spotter is located. Configure this in ⚙ Settings → DX Clusters → Spotter Region Filter.

Callsign Required for Spotting

Both C1 telnet spotting and Spothole spot submission require your callsign to be set in ⚙ Settings. Clusters use your callsign for the DX de YOURCALL header of each spot. Spothole uses it as the de_call field. If your callsign is not set, SDRLogger+ will show an error message rather than submit a blank spot.

Backup & Restore

SDRLogger+ stores all QSOs in two local SQLite database files: hamlog.db for general logging and pota.db for POTA activations. The Backup & Restore tools in ⚙ Settings → Backup & Restore operate on whichever database is currently active (based on your GENERAL / 🏕 POTA mode toggle).

Database Backup

Click Backup DB to save a timestamped copy of your QSO database (hamlog_backup_YYYYMMDD_HHMMSS.db).

  • If a Database Backup Folder path is configured (e.g. C:\Radio\Backups), the file is written directly to that folder and the full saved path is confirmed on screen.
  • If the folder field is left blank, the file downloads through your browser instead.
Set a backup folder path and click Save Settings first — then every future backup goes straight to that folder with one click.

Database Restore

Click Restore DB and pick a previously backed-up .db file. Before overwriting anything, SDRLogger+ automatically saves a safety copy of the current database (hamlog_pre_restore_YYYYMMDD_HHMMSS.db or pota_pre_restore_... in POTA mode) to your backup folder (or alongside the active database if no folder is configured). A ✅ Restore complete confirmation appears and the log panel reloads automatically.

Restore replaces your entire active QSO database. The automatic safety backup means you can always undo — but verify you are in the correct mode (General vs POTA) before restoring.

Settings Export & Import

Your SDRLogger+ configuration (callsign, QRZ credentials, cluster servers, rotator settings, display preferences, etc.) is stored in your browser's local storage. To back it up or move it to a different browser or machine:

  • Export Settings — downloads hamlog_settings_YYYYMMDD_HH.json containing all your preferences.
  • Import Settings — pick a previously exported .json file. All settings are restored and the page reloads automatically.
Export your settings after initial setup and save the file alongside your database backups. If your browser cache is ever cleared or you switch to a new PC, import the file and you are back up immediately — no re-entering QRZ keys or cluster servers.

🏕 POTA — Parks on the Air

SDRLogger+ includes a built-in POTA activation mode that adds dedicated park logging fields, P2P (Park-to-Park) contact tracking, POTA-ready ADIF export, and a one-click Spot Myself button to announce your activation to the POTA.app network.

Switching to POTA Mode

Click the GENERAL / 🏕 POTA toggle in the New QSO Entry panel header (right side of the panel title bar). When POTA mode is active:

  • A green 🏕 ACTIVATING: K-1234 banner appears above the QSO entry form.
  • An optional P2P Park Ref field appears in the QSO form — fill this in when the contact is also activating a park.
  • All contacts are saved to a separate pota.db database — your general log is completely unaffected.
  • The header log panel shows only your POTA contacts for the current session.
The GENERAL / 🏕 POTA toggle remembers your last mode between sessions. All other panels (Rotator, DX Cluster, HamLib, Distance Map) work exactly the same in both modes.

My Park Reference

Enter your activation park reference directly in the green activation banner (e.g. K-1234) — it persists for the entire session. You can also set a default park in ⚙ Settings → POTA so it pre-fills whenever you switch to POTA mode.

P2P (Park-to-Park) Contacts

When a hunter you work is also activating a park, enter their park reference in the P2P Park Ref field. SDRLogger+ automatically adds the ADIF fields SIG: POTA and SIG_INFO: K-5678 to that QSO. POTA awards P2P credit automatically when both activators upload matching logs — no extra steps needed.

Spot Myself (POTA Self-Spot)

Click the 📡 Spot Myself button in the activation banner to announce your activation to the POTA.app ↗ network so hunters can find you on the cluster. This uses the official POTA API and requires a free POTA.app account.

  1. Create a free account at pota.app ↗
  2. Enter your POTA.app username and password in ⚙ Settings → POTA
  3. Make sure your callsign and park reference are set
  4. In POTA mode, click 📡 Spot Myself — a confirm strip appears showing what will be sent
  5. Click SEND TO POTA — your spot goes to pota.app instantly
Frequency and mode are pulled automatically from TCI or HamLib if connected. You can also enter them manually in the QSO form before spotting.

POTA ADIF Export

Click ⬇ Export ADIF from the ADIF panel while in POTA mode. The exported file:

  • Is named pota_K-1234_YYYY-MM-DD.adi for easy identification
  • Includes MY_SIG: POTA and MY_SIG_INFO: K-1234 in every record
  • Includes SIG: POTA and SIG_INFO: K-5678 on P2P contacts
  • Is ready for direct upload to the POTA log submission page

Upload your log at: pota.app/#/user/logs/upload ↗

Offline Operation

POTA activations often happen in parks with no internet. SDRLogger+ works fully offline — simply disable QRZ lookup and DX cluster connections in Settings. TCI or HamLib will still provide frequency and mode. Only the Spot Myself button requires internet to reach the POTA API.

Applying Updates

SDRLogger+ includes a built-in zip updater so you can apply new versions without reinstalling. This is found in Settings → Server Settings & Software Update.

How to Apply an Update

  1. Obtain the update zip file (provided by the developer).
  2. Open Settings (⚙) and scroll to Server Settings & Software Update.
  3. Click 📦 Apply Update (upload zip) and select the zip file.
  4. SDRLogger+ will extract the files — your database is never touched (all .db files are automatically skipped).
  5. A full-screen overlay will appear confirming the update and showing restart instructions.

After Applying an Update

⚠ For best results, please restart your PC after applying an update.
  1. Close the browser tab.
  2. Stop SDRLogger+ — right-click the SDRLogger+ tray icon (taskbar, bottom-right) → Stop SDRLogger+.
  3. Close your other open applications.
  4. Restart your PC.
  5. After reboot — launch SDRLogger+ again (double-click the SDRLogger+ desktop shortcut or use the Start Menu).
  6. Open your browser to http://localhost:5000 (or the port you configured).
  7. Hard-refresh the browser: Ctrl + Shift + R
